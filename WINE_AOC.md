# Wine AOC prompt-context roadmap

## Purpose

Build a lightweight appellation-context layer for the Wine page so a clicked AOC
can produce a more specific and safer prompt in `app/utils/wine_prompts.py`.

This is a prompt-engineering and data-structuring task, not a geometry rewrite.
The map geometry and display labels can remain unchanged while we recover
semantic context from the raw INAO source and route prompts through a small,
explicit context model.

## Current status

### Application state

- The Wine map renders `assets/data/wine_regions_aoc.geojson`.
- The current app-level Wine data exposes:
  - `region`
  - `app`

- AOC clicks resolve through:


### Prompt state

`app/utils/wine_prompts.py` currently provides a region-aware prompt:

```python
generate_optimized_prompt(wine_region, appellation)
```

The function already distinguishes broad parent regions with:

```python
region_context = {
    "Bordeaux": {...},
    "Bourgogne": {...},
    "Loire": {...},
    ...

```

The next useful improvement is not a larger prose prompt. It is a local context
record that tells the prompt what kind of appellation was clicked before the API
call is made.

## Core principle

The cleaned map label is a display label, not the full legal or semantic
identity of the appellation.

For example:

```text
Raw source:  Alsace grand cru Altenberg de Bergheim
Map label:   Altenberg de Bergheim
Lost signal: Alsace Grand Cru
```

The map label should stay short, but the prompt context should preserve the
lost signal.

## Stage 1 — Profile raw and cleaned appellations

Create two unique-appellation tables.

### Raw profile

Minimum columns:

```text
raw_app
raw_app_norm
id_app_values
dt_values
categorie_values
type_denom_values
denom_values
```

Notes:

- Keep `dt` because it can help group source records, but do not treat it as a
  wine-origin hierarchy.
- Treat `categorie` as a style signal source, not as a classification authority.
- Keep source identifiers where available, but do not rely on them until their
  stability has been checked.

### Cleaned profile

Minimum columns:

```text
feature_id
display_app
display_app_norm
region
```

Do not use geometry except to resolve ambiguous name matches.

## Stage 2 — Build a raw-to-cleaned crosswalk

The crosswalk should preserve every raw source record that contributed to each
cleaned clickable AOC.

### Match types

Use a small controlled vocabulary:

```text

normalised
known_prefix_removed
alias_simplified
source_records_collapsed
raw_only
cleaned_only

```

### Examples


## Stage 8 — Prompt contract for `wine_prompts.py`

The application should resolve context locally before calling GPT-4.1 mini.

Target flow:

```python
context = resolve_wine_context(
    region=wine_region,
    appellation=appellation,
    feature_id=feature_id,
)

prompt = generate_optimized_prompt(
    wine_region=wine_region,
    appellation=appellation,
    context=context,
)
```

The model should not be asked to infer the appellation category from scratch on
every request.

### Prompt input contract

```python
WinePromptContext = {
    "display_name": str,
    "region": str,
    "canonical_names": list[str],
    "classification_profile": str,
    "style_modules": list[str],
    "prompt_modules": list[str],
    "facts": dict,
    "confidence": str,
}
```

### Prompt rules

The prompt should:

- focus on the clicked `appellation`;
- use `wine_region` as context, not as the main subject;
- include deterministic facts only when supplied by local context;
- ask for omission rather than invention when facts are absent;
- adapt emphasis from `classification_profile` and `style_modules`;
- keep the current restrained gastronomy-app tone.

## Stage 9 — Structured model output

```json

  "style_summary": "Concise account of wine style and structure.",
  "terroir_summary": "Relevant terroir explanation.",
  "pairings": [
```


The UI can later combine deterministic facts with generated interpretation.

## Immediate notebook task

Keep the next notebook pass narrow.

### Required outputs

1. Exact-match count between cleaned and raw appellations.
2. Cleaned-only appellation list.
3. Raw-only appellation list.
4. Automatic matches after:
   - punctuation normalisation;
   - apostrophe normalisation;
   - accent/case normalisation;
   - alias splitting around `ou`;
   - removal of known prefixes such as `Alsace grand cru`.
5. Cleaned appellations linked to multiple raw records.
6. Candidate routing dataframe.
7. Review CSV export.

### Initial success criterion

The pass is successful if most cleaned appellations can be linked to raw source
records automatically and the lost Alsace Grand Cru and Saint-Émilion context
can be retained without modifying map geometry.

## First deliverable

Create an appellation-context candidate table generated from the raw and cleaned
datasets.

It should answer:

> What source records contributed to this clickable appellation, what semantic
> information was lost during cleaning, and which broad context profile should
> it use?

That crosswalk becomes the foundation for later prompt routing, structured
facts, and modular UI rendering.
